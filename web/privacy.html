Privacy Policy

This site does not use marketing cookies, trackers, or analytics scripts. It operates an online compiler feature which stores limited technical data needed to run the service securely.

What We Collect & Store

When you submit code to the online compiler we create an execution record in our own SQLite databases located on the server. A single execution record currently contains:

  • container_id – internal sandbox identifier
  • created_at / finished_at – timestamps of start and end
  • execution_time_ms – total wall time in milliseconds
  • ip – your apparent client IP (extracted from X-Forwarded-For, X-Real-IP or the TCP connection)
  • code_executed – the full source code you submitted
  • output – combined stdout/stderr (truncated to 64KB)
  • error_message – error or timeout message if execution failed

Application logs are also stored in a separate SQLite database and may include:

  • ts – timestamp
  • level – log level (info, warn, error, etc.)
  • message – event description
  • caller / stack – source location and stack trace for errors

Rate limiting is enforced in-memory only; we keep a short‑lived token bucket per IP (not written to disk) for abuse prevention. No analytics profiling is performed.

How We Use This Data

  • Provide the compile/execute functionality and return results
  • Diagnose reliability or security issues (error logs / stack traces)
  • Investigate abuse (excessive usage, malicious code) using IP + code snippet

Third Parties

  • Google Fonts (fonts.googleapis.com / fonts.gstatic.com)
  • cdnjs.cloudflare.com (Prism.js)
  • GitHub (avatars/images)

These providers receive your IP to deliver content. No profiling or identification is intended by this site.

Data Retention

  • Execution history: records (code, IP, output, error) are automatically pruned after ~90 days.
  • Application logs: log entries (including stack traces) are also pruned after ~90 days.
  • In‑memory rate limit state: cleared after ~10 minutes of inactivity per IP.

We may delete data earlier during maintenance. Backups (if any) would follow the same retention window.

Online Compiler Inputs

Any code you submit ("Inputs") is stored exactly as provided in the execution history described above. We do not claim ownership; you retain all rights in your code. Do not submit proprietary or sensitive material.

  • Scope: Used only to compile/run and show you results, plus limited abuse review.
  • Visibility: Execution records are accessible only through authenticated admin endpoints.
  • Truncation: Output is truncated to protect resources; code limit is 5000 characters.

Please Avoid Personal Data

Do not submit personal data, credentials, or proprietary secrets in Inputs. The service is meant for code samples only.

Security & Sandbox

Inputs may run in isolated sandboxes with resource limits. No system is perfectly secure; results are provided on an as‑is basis.
